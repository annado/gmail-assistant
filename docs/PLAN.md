# Plan: Personal Gmail MCP Server

## Goal

Build a standalone Python MCP server that gives Claude Code direct Gmail access — starting with mark-as-read, fetch email body, and room to grow with more tools (archive, label, etc). This is a general-purpose Gmail MCP server, not tied to any specific skill.

## Approach: Test-Driven Development

Write tests first for each layer, then implement until they pass. Order optimized to enable real Gmail e2e testing as early as possible:

1. **gmail_auth** — OAuth flow. Tested with mocked filesystem/credentials.
2. **gmail_helpers** — pure logic (MIME parsing, HTML→markdown). Easy to unit test with no mocks.
3. **gmail_api** — API calls (mark as read, fetch). Tested with mocked Gmail service. **← e2e checkpoint: test against real Gmail in Python REPL**
4. **server** — MCP tool registration. Integration tested with FastMCP test client. **← e2e checkpoint: `fastmcp dev server.py` web inspector**
5. **gmail_safety** — logging, undo tracking. Layer `@logged` onto already-working tools.

### 2. Install `uv` and set up project

`uv` is the modern Python project manager (replaces pip, venv, pip-tools). It's fast and handles everything via `pyproject.toml`.

```bash
curl -LsSf https://astral.sh/uv/install.sh | sh
```

Then initialize the project:

```bash
cd gmail-mcp
uv init
```

This creates `pyproject.toml` and `.venv` automatically. Add `requires-python = ">=3.11"` to `pyproject.toml`. Then add dependencies:

```bash
uv add fastmcp google-api-python-client google-auth google-auth-oauthlib markdownify
uv add --dev pytest pytest-asyncio
```

Note: `google-auth` listed explicitly (not just transitively via `google-auth-oauthlib`) since we import from it directly. `pytest-asyncio` needed because MCP tools are `async def`.

After this, run anything with `uv run <command>` — no need to activate venvs.

Use FastMCP's built-in test client:
```python
from fastmcp.client import Client
from server import mcp

@pytest.mark.asyncio
async def test_mark_as_read_tool():
    async with Client(mcp) as client:
        result = await client.call_tool("gmail_mark_as_read", {"message_id": "abc123"})
```

**Then implement** (`server.py`):
```python
from fastmcp import FastMCP

mcp = FastMCP(name="Gmail")

@mcp.tool
async def gmail_mark_as_read(message_id: str) -> str:
    """Mark a Gmail message as read given its message ID. Returns confirmation with the email subject."""
    ...

@mcp.tool
async def gmail_fetch_body(message_id: str) -> str:
    """Fetch a Gmail message body as markdown given its message ID. Handles HTML newsletters and multipart MIME."""
    ...

if __name__ == "__main__":
    mcp.run()
```

FastMCP uses the docstring as the tool description and type hints as the parameter schema automatically. Use `Annotated[str, "description"]` for parameter-level docs if needed.

## Safety: Logging, Undo, and Guardrails

### Philosophy

Don't try to constrain the LLM with soft gates it can bypass (like `confirm=True` params). Instead, rely on layers that are actually enforceable:

1. **Activity log** (server-side, always written) — you always know what happened
2. **Undo log** (server-side) — reversible actions record their inverse
3. **Claude Code permission prompt** (harness-enforced) — user approves each tool call
4. **Don't expose dangerous tools** — if `gmail_delete` doesn't exist in the server, nobody can call it

### Activity log

Every tool call gets logged to `gmail-mcp/activity.jsonl` (JSON Lines, append-only). One line per action:

```json
{"timestamp": "2026-03-28T14:30:00Z", "tool": "gmail_mark_as_read", "args": {"message_id": "abc123"}, "subject": "L3 Field Trip Tomorrow!", "result": "success"}
```

This answers "what did Claude do to my email?" at any point.

### Descriptive return values

Every tool returns a human-readable string that includes the email subject, so Claude naturally relays it to you:

```
✓ Marked as read: "L3 Field Trip Tomorrow!" (abc123)
```

This is the simplest way to get console feedback — Claude shows you the tool's return value. No extra infrastructure needed for v1.

### Undo log

For modifying actions, record the inverse operation in the same activity.jsonl entry using the `UndoAction` dataclass (not a freeform dict):

```python
@dataclass
class UndoAction:
    action: str   # e.g. "add_label"
    label: str    # e.g. "UNREAD"
```

```json
{"timestamp": "...", "tool": "gmail_mark_as_read", "message_id": "abc123", "subject": "...", "result": "success", "undo": {"action": "add_label", "label": "UNREAD"}}
```

**Edge case:** If `gmail_mark_as_read` is called on an already-read message, the undo entry would incorrectly record "add UNREAD". Check the current label state before recording the inverse — only record undo if the message was actually unread.

This enables a future `gmail_undo_last` tool or manual recovery via a script.

### Error handling

Tools must return descriptive error strings, never crash the server. Catch errors at the `gmail_api` layer:

| Error | Behavior |
|---|---|
| Invalid/unknown `message_id` (404) | Return `"Error: message {id} not found"` |
| Expired/revoked token (`RefreshError`) | Auto-refresh once; if that fails, return `"Authentication expired. Re-run gmail_auth.py to re-authorize."` |
| Rate limiting (429) | Return `"Error: Gmail rate limit hit. Try again shortly."` |
| Network failure (`ConnectionError`) | Return `"Error: could not reach Gmail API. Check network."` |
| Other `HttpError` | Return `"Error: Gmail API returned {status}: {message}"` |

The `@logged` decorator should log failures too (with `"result": "error"`). Subject-fetch failure in the logger must not crash the tool — fall back to `"(subject unavailable)"`.

### Future: MCP Channels

For real-time push notifications independent of tool responses (e.g., background confirmations, batch operation progress), the MCP protocol supports **Channels** — server-initiated messages to the Claude Code terminal. This requires `claude --channels` and more setup. Not needed for v1, but the right path for v2 if we want richer feedback.

## Decisions made

- **`uv` over pip/venv** — modern standard, faster, handles lockfiles
- **TDD** — tests first for each module, then implement
- **Separate `gmail_helpers` from `gmail_api`** — keeps pure parsing logic testable without mocking API calls
- **`gmail.readonly` + `gmail.labels` scopes** — minimal privilege: readonly for fetching message bodies, labels for mark-as-read (removing UNREAD label). Avoids `gmail.modify` which would grant trash/send/etc. Re-auth with broader scopes when future tools require them
- **Own copy of all code** — no imports from school-emails or email_summarizer
- **Don't migrate school-emails** — leave `fetch_email_body.py` as-is for now
- **Copy credentials.json** — use existing Google OAuth client from assistantly project
- **No `confirm=True` gates** — LLMs can bypass soft parameter checks; rely on Claude Code's permission prompt (harness-enforced) + activity log (server-enforced) instead
- **Don't expose destructive tools until needed** — the best safety for `gmail_delete` is not having it exist yet
- **Descriptive return values** — tool returns include email subject so Claude naturally shows "✓ Marked as read: [subject]" to the user
- **Activity log with undo info** — every modifying action records its inverse for future recovery
- **Standalone `fastmcp` over `mcp[cli]`** — the standalone FastMCP package (`from fastmcp import FastMCP`) is more actively maintained than the SDK's copy (`mcp.server.fastmcp`). 1M+ downloads/day, powers ~70% of MCP servers. Docstrings become tool descriptions, type hints become parameter schemas automatically
- **Async tools** — MCP tools defined as `async def` since Gmail API calls are I/O-bound; avoids blocking the event loop
- **`@logged` inner, `@mcp.tool` outer** — decorator ordering matters: `@mcp.tool` inspects function metadata, so `@logged` must use `functools.wraps` and be the inner decorator
- **`UndoAction` dataclass over freeform dict** — typed undo metadata ensures future tools provide consistent undo info
- **Error strings, not exceptions** — tools return descriptive error messages on failure; never crash the MCP server
- **`.gitignore` first** — created before any credentials touch disk to prevent accidental secret commits
- **Credentials + auth early, safety last** — reordered to enable real Gmail e2e testing by step 6 (REPL) and step 7 (web inspector), rather than waiting until the end. Safety is a cross-cutting concern layered onto already-working tools

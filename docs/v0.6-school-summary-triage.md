# Improve School Email Summarization

_Depends on v0.5 validation harness (see `v0.5-validation-harness.md`). The harness encodes this plan's target format as fixtures + linter assertions, so v0.6 iterates against a reproducible red→green loop rather than live Gmail._

## Context

The current `/school-emails` skill produces a fresh summary file each run with four sections (Action Items, Calendar, Per-Child Notes, Emails Included). It works, but has two gaps for busy parents:

1. **Tone is descriptive, not executive.** The prompt asks for bullet points but doesn't push Claude to lead with the bottom line, surface what changed since last time, or hold back follow-up detail until asked.
2. **No memory across runs.** Every summary is independent. A field-trip permission slip mentioned three weeks running shows up three times. There is no way for the parent to say "don't care" or "already done" and have that stick.

The user wants the assistant to behave like a good EA: tell them what matters now, in one glance, and remember what they've already triaged.

This plan keeps the implementation lightweight — no new MCP tools, no JSON schemas, no database. The skill is Claude-driven, so we add state as a single human-editable markdown file and teach the skill to read/write it.

## Approach

### 1. Add a persistent triage log: `school-emails/school-tasks.md`

A single markdown file written and re-read by the skill every run. Claude maintains it; the parent can also hand-edit it.

**Location:** `./school-emails/school-tasks.md`, relative to the cwd where Claude Code was invoked (i.e. the calling project, e.g. `~/src/personal-assistant/school-emails/school-tasks.md`). All skill outputs — `school-tasks.md`, `school-tasks-archive.md`, and the dated `school-summary-YYYY-MM-DD.md` files — live in this same `school-emails/` subdirectory so the calling project doesn't accumulate loose files at its root. The skill creates the directory on first run.

**Format** (one section per status, simple checkbox-ish syntax so a human can flip status by editing one character):

```markdown
# School Tasks & Events — Running Log

_Last updated: 2026-04-07_

## Active

- [ ] **2026-04-10** Permission slip — Alex field trip to museum _(from: office@lincoln.edu, 2026-04-01)_
- [ ] **2026-04-15** Parent-teacher conference signup — Sam _(from: ms-davis@lincoln.edu, 2026-04-03)_
- [ ] **ongoing** Bring water bottle daily — Alex _(from: pe@lincoln.edu, 2026-04-02)_

## Done
- [x] **2026-04-05** Picture day order form — Alex ~~done 2026-04-04~~

## Don't care
- [~] Spring fundraiser auction — whole school ~~dismissed 2026-04-02~~
```

**Status markers** (chosen so a parent can edit by hand):
- `[ ]` active
- `[x]` done
- `[~]` don't care / dismissed

Each item carries: due date (or `ongoing`/`fyi`), short title, child tag, and a parenthetical provenance (sender + email date) so Claude can dedupe and the parent can trace it.

**Why a single markdown file (not JSONL, not a new MCP tool):**
- The skill is already Claude-driven; Claude can parse and rewrite markdown trivially.
- A parent can open it in any editor and flip `[ ]` → `[x]` without learning any tool.
- No code changes to `mcp/server.py` or `mcp/gmail_api.py` needed.
- Mirrors the project's existing pattern of writing freeform markdown summaries.

### 2. Rewrite `prompt.md` for executive-assistant tone

Replace the current four-section template with a leaner structure that leads with the bottom line and explicitly tells Claude to **withhold detail unless asked**.

New structure:

```markdown
## TL;DR
2-4 bullets. The single most important things this parent must know right now.
Lead with anything time-sensitive in the next 7 days. No fluff, no greetings,
no "the school sent...".

## This Week — Action Required
Checklist of things the parent must DO, with deadline in bold. Only items not
already in the Active triage log. Skip anything in Done or Don't care.

## Upcoming Calendar
Chronological dates within the next ~3 weeks. Group by date. Tag [Child Name].
Only events involving one of my children, school-wide closures, or parent
events. Skip generic activities that don't involve my children.

## Per-Child Notes
One short paragraph per child mentioned, ONLY if there is something
non-routine to flag (a teacher shoutout, a behavioral note, a heads-up about
a unit change). If there's nothing notable, omit the child entirely. No dates.

## Still On Your Plate
Compact one-line list of Active triage items NOT already surfaced above
(i.e. not urgent, not new this run). Format: `- [due date] title — [Child]`.
If there are more than 10, show the 10 closest deadlines and end with
`…and N more in school-tasks.md`. Omit the section entirely if Active is empty.

## Sources
Compact table: Date | Sender | Subject. One row per email parsed.
```

Style rules added to the prompt:
- **Lead, don't narrate.** No "The school sent an email about...". Just state the fact.
- **Withhold the long version.** Each bullet is one line. If there's backstory, end the bullet with "_(ask for details)_" instead of explaining inline.
- **Cut anything already triaged.** Items appearing in the Done or Don't care sections of `school-tasks.md` must not appear in the new summary at all.
- **Cut anything still pending from prior runs unless the deadline got closer or new info arrived.** If an item is already Active in the log and unchanged, mention it in the TL;DR only if it's now urgent (≤3 days).
- **Calendar is the only place dates live.** (Keep this rule from current prompt.)

### 3. Update `SKILL.md` workflow

Insert two new steps so the skill reads the triage log before summarizing and updates it after.

Updated steps:

1. Load personal context (`context.toml`) — unchanged.
2. **NEW: Load triage log.** Read `./school-emails/school-tasks.md` (relative to cwd) if it exists. Parse the three sections (Active / Done / Don't care). Hold them in working memory. If the `school-emails/` directory doesn't exist yet, create it.
3. Find school emails — unchanged.
4. Fetch email bodies — unchanged.
5. Read cached emails — unchanged.
6. Read `prompt.md` — unchanged.
7. **NEW: Reconcile against the triage log.** Before drafting the summary:
   - Drop any email-derived task that matches an item in Done or Don't care (match on title + child, fuzzy).
   - For tasks already in Active, don't re-list unless the deadline is now ≤3 days or the email contains genuinely new info.
   - Collect any brand-new tasks/events to add to the log.
8. Write the summary file (`./school-emails/school-summary-YYYY-MM-DD.md`) following the new prompt.
9. **NEW: Update `./school-emails/school-tasks.md`.**
   - Append new active items to the Active section.
   - Update `_Last updated:_` line.
   - Move any items the parent told the assistant about ("I already did X", "don't care about Y") into the appropriate section. (The skill description should mention the parent can say this in chat.)
   - Auto-archive: items in Active whose due date is more than 7 days in the past get moved to Done with a note `~~auto-archived YYYY-MM-DD~~`.
   - **Roll over old triage history.** To keep the file scannable, any item in Done or Don't care older than 30 days gets moved out of `school-tasks.md` and appended to `school-tasks-archive.md` (created on demand). The archive is write-only from the skill's perspective — it's never re-read for dedupe; instead, when reconciling in step 7, the skill fuzzy-matches against both files but only loads the archive if the active log misses. In practice, anything 30+ days old is unlikely to recur, and if it does the parent can re-triage.
   - **Cap the Active section.** If Active grows past ~25 items, the skill flags this in the next summary's TL;DR ("Triage backlog: 27 active items — consider sweeping `school-tasks.md`") so the parent knows to prune. The skill never silently drops Active items.
10. Mark as read (optional) — unchanged.

### 4. Document the user-facing interaction

Add a short "How to triage" section near the top of `SKILL.md` so the parent knows how to talk to it:

> After running `/school-emails`, you can tell the assistant things like:
> - "Mark the field trip slip as done"
> - "Don't care about the fundraiser auction"
> - "What's the backstory on the science fair?" (the summary intentionally omits detail — ask and the assistant will pull it from the cached email)
>
> You can also edit `school-emails/school-tasks.md` directly — flip `[ ]` to `[x]` (done) or `[~]` (don't care) and the next summary will respect it.

## Files to Modify

- **`skills/school-emails/prompt.md`** — full rewrite with TL;DR-first structure and executive-assistant style rules (current file: 20 lines).
- **`skills/school-emails/SKILL.md`** — add steps 2, 7, 9; add "How to triage" section; `allowed-tools` is unchanged (already includes Read/Write/Glob).

## Files NOT Modified (and why)

- `mcp/server.py`, `mcp/gmail_api.py`, `mcp/gmail_cache.py` — no new tools needed; the skill is pure Claude + Read/Write/Glob.
- `context.toml` / `context.example.toml` — children/teacher metadata is orthogonal to triage state.
- No new tests — the change is in skill prompts and a markdown file format, not Python code.

## Verification

1. **Dry run with existing inbox.** Run `/school-emails`. Confirm:
   - The output starts with a TL;DR section.
   - Each bullet is one line; no paragraphs of backstory.
   - `./school-emails/school-tasks.md` is created with active items populated, and the `school-emails/` directory is created if it didn't exist.
2. **Triage round-trip.** Tell the assistant "mark the permission slip as done" and "don't care about the auction". Confirm:
   - `school-tasks.md` shows those items moved to Done / Don't care.
   - Run `/school-emails` again on the same inbox; the dismissed items do NOT reappear in the new summary.
3. **Hand-edit round-trip.** Manually flip an `[ ]` to `[x]` in `school-tasks.md`. Re-run the skill. Confirm the item stays archived and is not re-suggested.
4. **Auto-archive.** Add a fake item with a due date 10 days in the past to the Active section. Run the skill. Confirm it's moved to Done with an `auto-archived` note.
5. **Follow-up question.** Ask "what was the science fair email about?" after a summary. Confirm the assistant pulls detail from the cached email file rather than re-summarizing everything.
6. **Long-term scaling.** Seed `school-tasks.md` with a Done item dated 45 days ago. Run the skill. Confirm the item is moved to `school-tasks-archive.md` and `school-tasks.md` no longer contains it. Seed Active with 26 items and confirm the next summary's TL;DR includes a "Triage backlog" warning.

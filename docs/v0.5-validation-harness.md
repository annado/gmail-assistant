# School Email Skill — Validation Harness

## Context

The `/school-emails` skill is prompt-driven (Claude + Read/Write/Glob, no custom Python), so normal unit tests don't apply. Today, iterating on `prompt.md` means re-running the live skill against real Gmail and eyeballing the output — slow, non-reproducible, and hard to diff.

v0.5 fixes that before any prompt change. It ships a fixture folder, a structural linter, and an on-demand eval driver. All three encode the _target_ v0.6 behavior (see `v0.6-school-summary-triage.md`) as assertions. Running the harness against the current, unchanged `prompt.md` produces a red baseline — that's expected, and v0.6 turns it green.

**Non-goal in v0.5:** changing any skill behavior. `prompt.md`, `SKILL.md`, `context.toml`, and `mcp/*` are untouched. v0.5 ships standalone and is useful on its own as a specification of the target format.

## Approach

### 1. Fixture folder: `skills/school-emails/fixtures/`

One subdirectory per scenario. Each scenario is self-contained:

```
skills/school-emails/fixtures/
  scenario_new_items/        — brand-new tasks, no prior log
  scenario_dup_of_active/    — email restates an Active item (should not re-surface unless urgent)
  scenario_dup_of_done/      — email mentions a Done item (should be dropped entirely)
  scenario_dup_of_dontcare/  — email mentions a Don't care item (should be dropped entirely)
  scenario_stale_active/     — Active item 10 days past due (should auto-archive to Done)
  scenario_rollover/         — Done item 45 days old (should roll over to school-tasks-archive.md)

  scenario_*/
    context.toml      — fake children metadata (Alex + Sam, no real PII)
    tasks_before.md   — prior triage state (optional; absent = first run)
    emails/*.md       — 3–5 fake cached emails matching the emails/YYYY/MM/{id}.md schema
    expect.yaml       — rubric for --judge mode, e.g.
                          asserts:
                            - TL;DR does not mention the picture day form
                            - Still On Your Plate lists the water bottle item
```

Fixture emails mirror the real cache schema from `mcp/gmail_cache.py` (YAML frontmatter with `id`, `subject`, `from`, `date`, followed by markdown body) so the same Read-tool workflow applies. Fixtures are checked in (no real PII) and serve as the single source of truth for both manual replay (paste into a Claude Code session) and scripted eval.

### 2. Structural linter: `scripts/check_school_outputs.py`

Zero-API Python script (~80 lines). Parses a generated `school-summary-*.md` and a `school-tasks.md` pair and asserts the target v0.6 format. Exit non-zero on any failure.

**Summary checks:**
- Sections appear in prescribed order: TL;DR → This Week — Action Required → Upcoming Calendar → Per-Child Notes → Still On Your Plate (optional) → Sources.
- TL;DR has ≤4 bullets.
- No bullet exceeds 200 chars.
- Every date string appearing anywhere in the file also appears in the Upcoming Calendar section (enforces "calendar is the only place dates live").
- Sources table has columns `Date | Sender | Subject`.

**Tasks-file checks:**
- Only `[ ]` / `[x]` / `[~]` status markers used.
- Every due date is parseable (ISO date) or one of `ongoing` / `fyi`.
- No item text appears in two sections simultaneously.
- `_Last updated:_` line present and parseable.

Callable standalone: `python scripts/check_school_outputs.py <summary.md> <tasks.md>`. Also imported by the eval driver.

**Expected v0.5 behavior:** running the linter against the current skill's output will fail (no TL;DR section, no `school-tasks.md` file at all). That's the red baseline — the v0.5 PR should document it explicitly so the failure is expected, not a regression.

### 3. On-demand eval driver: `scripts/eval_school.py`

~150-line Python script that replays fixture scenarios against the real Claude API. Run manually during prompt-tuning sessions only. No CI, no cron, no snapshot diffing.

**Per-scenario behavior:**

1. Load `context.toml`, `tasks_before.md` (if present), all email fixtures from `scenario_*/emails/`.
2. Build one user message inlining: children context, prior tasks log, all email bodies, plus the literal contents of `skills/school-emails/prompt.md`. Ask the model to emit two fenced blocks: ` ```summary ` and ` ```tasks-after `.
3. Call `client.messages.create(model="claude-sonnet-4-6", temperature=0, max_tokens=2000)`.
4. Parse the two fenced blocks; write each to `tmp/eval-out/<scenario>/` for human inspection.
5. Run `check_school_outputs.py` inline on the pair; print colored pass/fail.
6. If `--judge`: second call to `claude-haiku-4-5` with the scenario's `expect.yaml` rubric, asking for structured JSON `{pass: bool, failures: [str]}`. Print result.
7. Aggregate exit code across scenarios.

**Flags:**
- `scripts/eval_school.py` — all scenarios, linter only.
- `scripts/eval_school.py <scenario_name>` — one scenario.
- `scripts/eval_school.py --judge` — add LLM-as-judge rubric checks.
- `scripts/eval_school.py --lint-only` — skip the API; re-lint whatever is in `tmp/eval-out/`. Useful for iterating on the linter without spending tokens.

**Explicit non-goals:**
- No snapshot diffing. Human eyeballs the `tmp/eval-out/` files. Solo dev, not worth the churn.
- No CI integration.
- No nightly runs. Zero cost when not actively tuning.

**Cost envelope** (Sonnet 4.6 generator, Haiku 4.5 judge, ~5K-token scenarios):

| Activity | Cost |
|---|---|
| One scenario, linter only | ~$0.04 |
| Full 6-scenario run, linter only | ~$0.25 |
| Full run with `--judge` | ~$0.37 |
| Typical tuning session (~15 full runs) | ~$4 |
| Quiet month (no tweaks) | $0 |

Env: `ANTHROPIC_API_KEY` read from environment. No key committed.

## Verification

1. **Linter standalone.** Hand-craft a fake summary/tasks pair matching the target format; run `python scripts/check_school_outputs.py` against it — expect pass. Break one rule (e.g. add a fifth TL;DR bullet), expect fail with a clear diagnostic.
2. **Eval driver smoke test.** Run `python scripts/eval_school.py scenario_new_items`. Confirm the script calls the API, writes `tmp/eval-out/scenario_new_items/summary.md` and `tasks-after.md`, invokes the linter, and reports a result.
3. **Red baseline documented.** Run `python scripts/eval_school.py` against the unmodified current `prompt.md`. Expect most scenarios to fail the linter (no TL;DR, no `tasks-after` block). Capture the output in the v0.5 PR description as "expected baseline; v0.6 turns this green."
4. **Lint-only mode.** Re-run `scripts/eval_school.py --lint-only` and confirm the existing `tmp/eval-out/` files are re-checked without an API call.
5. **Judge smoke test.** Run `scripts/eval_school.py scenario_new_items --judge`. Confirm the judge call succeeds and returns structured JSON. (Content failures are fine — we're testing the plumbing.)

## Sequencing Notes

- v0.5 merges cleanly even if v0.6 never happens — the fixtures and scripts are useful as documentation of the target design.
- The v0.5 PR description should include a paste of the red-baseline eval output so reviewers understand the linter is intentionally failing against current behavior.
- If v0.6 reveals linter rules that are too strict (e.g. Claude legitimately wants a section reordered), update the linter in v0.6 — the linter isn't sacred, it's a tool for iteration.

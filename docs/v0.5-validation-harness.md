# School Email Skill — Validation Harness

## Context

The `/school-emails` skill is prompt-driven (Claude + Read/Write/Glob, no custom Python), so normal unit tests don't apply. Today, iterating on `prompt.md` means re-running the live skill against real Gmail and eyeballing the output — slow, non-reproducible, and hard to diff.

v0.5 fixes that before any prompt change. It ships a fixture folder, a structural linter, and an on-demand eval driver. All three encode the _target_ v0.6 behavior (see `v0.6-school-summary-triage.md`) as assertions. Running the harness against the current, unchanged `prompt.md` produces a red baseline — that's expected, and v0.6 turns it green.

**Non-goal in v0.5:** changing any skill behavior. `prompt.md`, `SKILL.md`, `context.toml`, and `mcp/*` are untouched. v0.5 ships standalone and is useful on its own as a specification of the target format.

## Approach

Everything for v0.5 lives under `skills/school-emails/eval/`. This keeps the harness colocated with the skill it tests, distinct from `mcp/tests/` (pytest unit tests), and self-contained so a future second skill can add its own `eval/` dir without cross-contamination.

```
skills/school-emails/eval/
  fixtures/
    new_items/
    dup_of_active/
    dup_of_done/
    dup_of_dontcare/
    stale_active/
    rollover/
  check_school_outputs.py    — structural linter + diff helpers
  eval_school.py             — on-demand eval driver (SKILL.md emulator)
```

### 1. Fixtures: `skills/school-emails/eval/fixtures/`

One subdirectory per scenario. Each scenario is self-contained:

```
fixtures/
  new_items/         — brand-new tasks, no prior log
  dup_of_active/     — email restates an Active item (should not re-surface unless urgent)
  dup_of_done/       — email mentions a Done item (should be dropped entirely)
  dup_of_dontcare/   — email mentions a Don't care item (should be dropped entirely)
  stale_active/      — Active item 10 days past due (should auto-archive to Done)
  rollover/          — Done item 45 days old (should roll over to school-tasks-archive.md)

  <scenario>/
    context.toml             — fake children metadata (Alex + Sam, no real PII)
    tasks_before.md          — prior triage state (optional; absent = first run)
    emails/*.md              — 3–5 fake cached emails matching the emails/YYYY/MM/{id}.md schema
    expect.yaml              — assertions for linter + judge (schema below)
    tasks_after_golden.md    — optional hand-written "ideal" tasks-after for diff inspection
```

Fixture emails mirror the real cache schema from `mcp/gmail_cache.py` (YAML frontmatter with `id`, `subject`, `from`, `date`, followed by markdown body) so the same Read-tool workflow applies. Fixtures are checked in (no real PII) and serve as the single source of truth for both manual replay (paste into a Claude Code session) and scripted eval.

#### `expect.yaml` schema

The linter handles the deterministic asserts; the judge (in `--judge` mode) handles `judge_rubric` only. Clean split, no overlap.

```yaml
summary:
  must_contain:                 # case-insensitive substring
    - "water bottle"
  must_not_contain:
    - "picture day form"
  section_asserts:
    "TL;DR":
      max_bullets: 4
      must_mention: ["field trip"]
    "Still On Your Plate":
      must_list: ["water bottle daily"]

tasks_after:
  active_must_contain: ["permission slip"]   # match on item title substring
  done_must_contain: ["picture day"]
  dontcare_must_contain: []
  must_not_appear_anywhere: []               # items that should be fully dropped

tasks_archive_append:                         # optional
  must_contain: []

judge_rubric:                                 # free-form, --judge mode only
  - "Does the TL;DR lead with the most time-sensitive item?"
  - "Are any dates mentioned outside the Upcoming Calendar section?"
```

### 2. Structural linter: `skills/school-emails/eval/check_school_outputs.py`

Zero-API Python script (~150–200 lines — initial 80-line estimate was optimistic given markdown parsing, date extraction, and useful diagnostics). Parses a generated `school-summary-*.md` and a `school-tasks.md` pair and asserts the target v0.6 format. Exit non-zero on any failure.

**Summary checks:**
- Sections appear in prescribed order: TL;DR → This Week — Action Required → Upcoming Calendar → Per-Child Notes → Still On Your Plate (optional) → Sources.
- TL;DR has ≤4 bullets.
- No bullet exceeds 200 chars.
- Every date string appearing anywhere in the file also appears in the Upcoming Calendar section (enforces "calendar is the only place dates live").
- Sources table has columns `Date | Sender | Subject`.

**Tasks-file checks:**
- Only `[ ]` / `[x]` / `[~]` status markers used.
- Every due date is parseable (ISO date) or one of `ongoing` / `fyi`.
- No item text appears in two sections simultaneously.
- `_Last updated:_` line present and parseable.

**`expect.yaml` checks:** consume `summary.*` and `tasks_after.*` blocks, walk the parsed structure, fail with a per-assertion diagnostic.

**Diff helpers (non-snapshot regression signal):**

The same module exposes two diff functions, used by the eval driver to compare a current run against a previous one without strict snapshot diffing:

- `shape_diff(summary_a, summary_b)` — extracts a normalized "shape" from each summary (section headings, bullet counts per section, dates in calendar, item titles — no prose) and prints a unified diff. Catches "TL;DR went from 3 to 7 bullets" or "an item disappeared" without flagging phrasing churn.
- `tasks_set_diff(tasks_a, tasks_b)` — parses both task files into sets of `(status, title, due_date)` tuples and prints added / removed / status-changed items. This is the highest-signal regression check because the tasks log is more structured than the summary, and the v0.6 reconcile/rewrite logic mutates it on every run.

Both diffs are deterministic, zero-API, and ~60 lines combined.

Callable standalone:
```
python -m skills.school_emails.eval.check_school_outputs <summary.md> <tasks.md>
python -m skills.school_emails.eval.check_school_outputs --diff <prev-dir> <curr-dir>
```

**Expected v0.5 behavior:** running the linter against the current skill's output will fail (no TL;DR section, no `school-tasks.md` file at all). That's the red baseline — the v0.5 PR should document it explicitly so the failure is expected, not a regression.

### 3. On-demand eval driver: `skills/school-emails/eval/eval_school.py`

~200-line Python script that replays fixture scenarios against the real Claude API. Run manually during prompt-tuning sessions only. No CI, no cron, no strict snapshot diffing.

**The driver is a SKILL.md emulator, not a prompt-only harness.** It loads both `SKILL.md` and `prompt.md` from the skill dir and inlines them into the user message, so the model executes the full v0.6 workflow (load tasks log → reconcile → write summary → rewrite tasks log → emit archive append). This means v0.5 tests `SKILL.md` + `prompt.md` as a pair, which is what we actually want — the reconcile/rewrite logic is the most likely regression site, and a pure-prompt harness would miss it.

Trade-off: ~20% more tokens per call vs. a prompt-only harness. Still cheap (~$5/tuning session vs. $4).

**Per-scenario behavior:**

1. Load `SKILL.md`, `prompt.md`, and the fixture's `context.toml`, `tasks_before.md` (if present), and all email fixtures from `<scenario>/emails/`.
2. Build one user message inlining: "You are running the school-emails skill. Here is your SKILL.md workflow: `<inlined>`. Here is `prompt.md`: `<inlined>`. Here is the context, the prior `school-tasks.md` (or 'none — first run'), and the cached email bodies. Execute the workflow and emit three fenced blocks: ` ```summary `, ` ```tasks-after `, ` ```tasks-archive-append ` (the last may be empty)."
3. Call `client.messages.create(model="claude-sonnet-4-6", temperature=0, max_tokens=4000)`.
4. Parse the three fenced blocks; write each to `tmp/eval-out/<scenario>/` for human inspection. Stamp `meta.json` with prompt SHA, SKILL.md SHA, model, timestamp.
5. Run `check_school_outputs.py` inline on the trio; print colored pass/fail.
6. If `--judge`: call `claude-haiku-4-5` with the scenario's `judge_rubric`, asking for structured JSON `{pass: bool, failures: [str]}`. Run the judge **3× and majority-vote** to absorb LLM-judge variance even at temp 0. Print result.
7. If `--diff <prev-dir>`: run `shape_diff` and `tasks_set_diff` against the previous run. Print added / removed / changed.
8. Aggregate exit code across scenarios.

**Flags:**
- `eval_school.py` — all scenarios, linter only.
- `eval_school.py <scenario_name>` — one scenario.
- `eval_school.py --judge` — add LLM-as-judge rubric checks (3× majority vote).
- `eval_school.py --lint-only` — skip the API; re-lint whatever is in `tmp/eval-out/`. Useful for iterating on the linter without spending tokens.
- `eval_school.py --out-dir <path>` — write outputs somewhere other than `tmp/eval-out/`. Lets you keep a "before" run around for diffing.
- `eval_school.py --diff <prev-out-dir>` — after running, compare against a previous run using shape diff + tasks-set diff.
- `eval_school.py --model <id>` — override the generator model (default `claude-sonnet-4-6`).

**Typical iteration loop:**

```bash
# capture baseline
eval_school.py --out-dir tmp/eval-before/

# edit prompt.md, SKILL.md, etc.

# compare
eval_school.py --out-dir tmp/eval-after/ --diff tmp/eval-before/
```

**Explicit non-goals:**
- No strict snapshot diffing. Shape diff + tasks-set diff give a regression signal without locking in phrasing.
- No CI integration.
- No nightly runs. Zero cost when not actively tuning.

**Cost envelope** (Sonnet 4.6 generator, Haiku 4.5 judge ×3, ~6K-token scenarios — slightly higher than original estimate due to inlining SKILL.md + 3× judge calls):

| Activity | Cost |
|---|---|
| One scenario, linter only | ~$0.05 |
| Full 6-scenario run, linter only | ~$0.30 |
| Full run with `--judge` (3× vote) | ~$0.50 |
| Typical tuning session (~15 full runs) | ~$5 |
| Quiet month (no tweaks) | $0 |

Env: `ANTHROPIC_API_KEY` read from environment. No key committed.

### 4. Fixture self-test (catch fixture rot)

A linter rule change can silently invalidate all fixtures. Add a `--check-fixtures` mode that runs the linter against each scenario's `tasks_after_golden.md` (where present) and expects pass. Run manually after editing the linter or fixtures. ~20 lines, no API calls.

## Files to Add

- `skills/school-emails/eval/fixtures/<scenario>/` — 6 scenario folders with fixtures (checked in; no real PII).
- `skills/school-emails/eval/check_school_outputs.py` — structural linter + shape diff + tasks-set diff.
- `skills/school-emails/eval/eval_school.py` — on-demand eval driver (SKILL.md emulator).
- `skills/school-emails/eval/__init__.py` — empty, makes the dir importable.
- `.gitignore` — add `tmp/eval-out/`, `tmp/eval-before/`, `tmp/eval-after/`.

## Files NOT Modified (and why)

- `skills/school-emails/prompt.md`, `skills/school-emails/SKILL.md` — v0.5 is pure harness; behavior changes happen in v0.6.
- `mcp/server.py`, `mcp/gmail_api.py`, `mcp/gmail_cache.py` — no new MCP tools needed.
- `context.toml` / `context.example.toml` — fixtures carry their own fake context.
- `mcp/tests/` — pytest unit tests are unchanged. The eval harness is deliberately separate so `pytest` doesn't accidentally run paid API calls.

## Verification

1. **Linter standalone.** Hand-craft a fake summary/tasks pair matching the target format; run the linter — expect pass. Break one rule (e.g. add a fifth TL;DR bullet), expect fail with a clear diagnostic.
2. **Fixture self-test.** Run `eval_school.py --check-fixtures`. Expect every scenario's `tasks_after_golden.md` to lint clean.
3. **Eval driver smoke test.** Run `eval_school.py new_items`. Confirm the script calls the API, writes `tmp/eval-out/new_items/summary.md`, `tasks-after.md`, and `tasks-archive-append.md` (possibly empty), invokes the linter, and reports a result.
4. **Red baseline documented.** Run `eval_school.py` against the unmodified current `prompt.md` + `SKILL.md`. Expect most scenarios to fail the linter (no TL;DR, no `tasks-after` block). Capture the output in the v0.5 PR description as "expected baseline; v0.6 turns this green."
5. **Lint-only mode.** Re-run `eval_school.py --lint-only` and confirm the existing `tmp/eval-out/` files are re-checked without an API call.
6. **Diff mode.** Copy `tmp/eval-out/` to `tmp/eval-before/`, re-run with `--out-dir tmp/eval-after/ --diff tmp/eval-before/`. Confirm shape diff + tasks-set diff print (probably empty, since nothing changed).
7. **Judge smoke test.** Run `eval_school.py new_items --judge`. Confirm the judge call succeeds, runs 3×, and returns a majority-voted result. (Content failures are fine — we're testing the plumbing.)

## Sequencing Notes

- v0.5 merges cleanly even if v0.6 never happens — the fixtures and scripts are useful as documentation of the target design.
- The v0.5 PR description should include a paste of the red-baseline eval output so reviewers understand the linter is intentionally failing against current behavior.
- If v0.6 reveals linter rules that are too strict (e.g. Claude legitimately wants a section reordered), update the linter in v0.6 — the linter isn't sacred, it's a tool for iteration.

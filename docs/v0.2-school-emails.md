# Plan: v0.2 — School Email Summarizer

## Context

The project is a Gmail MCP server (`/home/user/gmail-assistant`) that currently exposes two tools to Claude:
- `gmail_mark_as_read` — removes UNREAD label
- `gmail_fetch_body` — fetches email body as markdown

This iteration (v0.2) adds school email summarization. Future iterations will add order email processing and ad cleanup (see full roadmap plan for details).

## Scope: School Emails Only

Two new MCP tools added to `server.py`:
- `gmail_list_messages(query, max_results)` — find emails by Gmail search query
- `gmail_summarize_school_emails(message_ids)` — returns all email bodies + personal context so Claude can write one holistic summary

Claude does the summarization. MCP tools fetch/format data. No Claude API calls inside the MCP server.

## Module Details

### `context.toml` (user-created, git-ignored)

Personal context only — no templates here. Uses `[[children]]` array with flat teacher strings. A `context.example.toml` is committed to the repo as a template.

```toml
[[children]]
name = "Alex"
grade = 7
school = "Lincoln Middle School"
sports = ["soccer", "swim team"]
activities = ["robotics club"]
teachers = [
    "Ms. Johnson (Math)",
    "Mr. Lee (Science)",
]

# Add more children by repeating [[children]]:
# [[children]]
# name = "Sam"
# grade = 4
```

Supports multiple children. Also accepts singular `[child]` table for backwards compatibility. File is optional — if absent, school tool returns a graceful "No personal context configured" message following the existing "return strings not raise" pattern.

### `prompts/school_summary.md` (committed to repo)

Markdown file containing the summary template Claude must follow. One file per workflow type — future workflows (orders, ads) get their own files.

Example:
```markdown
Lead with any action items (deadlines, permission slips, things I need to do).
Then cover events and dates.
Note anything specific to Alex (by grade level or team).
Keep it concise — a few bullet points is ideal.
```

`gmail_school.py` reads this file at call time (resolved via `Path(__file__).parent / "prompts" / ...` so it works regardless of cwd) and includes its contents in the output under `SUMMARY INSTRUCTIONS`. If the file is absent, the section is omitted and Claude uses its default judgment.

### `context_config.py`

Frozen dataclasses: `ChildContext`, `PersonalContext`. Teachers are flat strings (e.g. `"Ms. Johnson (Math)"`). Single public function:
```python
def load_context(path: str | Path = "context.toml") -> PersonalContext
```

Uses `tomllib` (Python 3.12 stdlib — no new dependency). Returns `PersonalContext(children=())` if file is absent or malformed. Supports both `[[children]]` array and singular `[child]` table.

### `gmail_list.py`

```python
def list_messages(service, query: str, max_results: int = 10) -> str
```
Returns formatted lines: `"{message_id} | {subject} | {sender} | {date}"`. Claude calls this first to find candidate email IDs, then passes them to feature-specific tools. Date is included so Claude can judge recency when selecting emails to summarize.

### `gmail_school.py`

```python
def fetch_school_emails_with_context(
    service,
    message_ids: list[str],
    context: PersonalContext,
) -> str
```

Takes multiple message IDs so Claude can write one holistic summary covering all recent school emails at once (e.g. a school-wide email + two class emails received the same day).

Returns a structured string:
- `PERSONAL CONTEXT` — each child's name, grade, level, school, teachers, sports/activities (printed once)
- `SUMMARY INSTRUCTIONS` — contents of `prompts/school_summary.md` (omitted if file absent)
- For each message: `EMAIL N of M — {subject} from {sender} on {date}` followed by its markdown body

Read-only. Claude uses the returned data to write one combined parent summary, following the template if present.

### `server.py` additions

```python
@mcp.tool
async def gmail_list_messages(query: str, max_results: int = 10) -> str:
    """Search Gmail and return matching messages as 'id | subject | sender' lines.
    Accepts standard Gmail search syntax (e.g. 'from:school.edu is:unread')."""

@mcp.tool
async def gmail_summarize_school_emails(message_ids: list[str]) -> str:
    """Fetch multiple school emails with personal context (child grade, teachers, activities).
    Pass all related school emails together so Claude can write one holistic parent summary."""
```

Each tool follows existing pattern: `get_credentials` → `get_service` → delegate to sync function → return string.

## Gmail API Scopes

No changes needed. Existing `gmail.readonly` and `gmail.modify` scopes cover:
- `messages.list()` — search/filter
- `messages.get()` — fetch metadata or full body

## Typical Claude Workflow

1. `gmail_list_messages(query="from:school.edu is:unread")`
2. `gmail_summarize_school_emails(message_ids=["id1", "id2", "id3"])` — all at once
3. Claude writes one holistic parent-friendly summary covering all emails

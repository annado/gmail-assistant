# Plan: v0.4 — Skills Restructure

## Context

The project currently bundles everything into one MCP server: Gmail actions (mark as read, fetch body), search/listing, caching, and the school email summarization workflow. The summarization logic (personal context loading, prompt instructions, orchestrating cache + file paths) lives in MCP tool code, but it's really a Claude workflow — not a programmatic action. This restructure separates concerns: the MCP server handles Gmail actions, and Claude Code skills handle email management workflows.

## Background

### Email caching (v0.3)

The v0.3 release (see `docs/v0.3-email-cache.md`) introduced `gmail_cache.py` to solve MCP size limits. Emails are saved as local markdown files in `emails/YYYY/MM/{message_id}.md` with YAML frontmatter (id, subject, from, date) + body. Key functions:
- `save_email(service, message_id) -> Path` — fetch from Gmail API if not cached, write to disk
- `ensure_cached(service, message_ids) -> list[Path]` — batch cache, returns file paths
- `find_cached(message_id) -> Path | None` — glob lookup
- `read_cached(message_id) -> str | None` — read body (strips frontmatter)

Both `gmail_fetch_body` (MCP tool) and the skills rely on this cache. The `gmail_fetch_body` tool calls `save_email` then `read_cached` transparently.

> Note: the original v0.4 design intended skills to use cached file paths and Read the files directly, but the current implementation has skills receive the body string back through `gmail_fetch_body`. See `docs/TODOs.md` for the deferred refactor.

### Email markdown cleanup (v0.2b — not yet implemented)

`docs/v0.2b-email-cleanup.md` plans an HTML preprocessing step in `gmail_helpers.py` to shrink cached emails from ~24KB to ~11KB. Not blocking this restructure — handled separately.

## Architecture

### MCP server (gmail actions only)

All MCP code lives in `mcp/`. Tools that do things to Gmail that Claude can't do natively:
- `gmail_mark_as_read` — mark message read
- `gmail_fetch_body` — fetch + cache email body as markdown
- `gmail_list_messages` — search Gmail by query
- (Future) `gmail_archive` — archive a message
- (Future) `gmail_delete` — trash a message

Removed `gmail_summarize_school_emails` — that workflow moved to a skill.

#### Running the MCP server

`.mcp.json` uses `--directory` pointing at `mcp/` and runs `python server.py`. Credential files (`token.json`, `credentials.json`) live alongside the server in `mcp/`, so relative paths just work.

```json
{
  "mcpServers": {
    "gmail-assistant": {
      "command": "uv",
      "args": ["run", "--directory", "/Users/anna/src/gmail-assistant/mcp", "python", "server.py"]
    }
  }
}
```

**Important:** Any project that uses this MCP server (e.g. `personal-assistant`) must copy this same config into its own `.mcp.json`.

#### EMAILS_DIR path

`gmail_cache.py` defines `EMAILS_DIR = Path(__file__).resolve().parent.parent / "emails"` — resolves from `mcp/` up to the project root's `emails/` directory, so cached emails live at the project root regardless of which working directory invokes the server.

#### Test configuration

`pyproject.toml` sets `pythonpath = ["mcp"]` so pytest can resolve bare imports (`from gmail_auth import ...`) when run from the project root:
```bash
uv run python -m pytest mcp/tests/ -v
```

### Skills (email management workflows)

#### `/school-emails` skill
Orchestrates: search school emails → fetch bodies → summarize with personal context.

```
skills/school-emails/
├── SKILL.md              # Skill definition + instructions
├── prompt.md             # Summary formatting rules
├── context.toml          # Children, grades, teachers, etc. (git-ignored)
└── context.example.toml  # Template for new users
```

SKILL.md frontmatter:
```yaml
---
name: school-emails
description: Summarize recent school emails with personal context about your children
allowed-tools: "mcp__gmail-assistant__gmail_list_messages mcp__gmail-assistant__gmail_fetch_body mcp__gmail-assistant__gmail_mark_as_read Read Write Glob"
---
```

SKILL.md body instructs Claude to:
1. Read skill-local `context.toml` for personal context
2. Build a single OR-query per school from `school_from` addresses and call `gmail_list_messages`
3. Call `gmail_fetch_body` for each message
4. Read `prompt.md` for summary formatting rules
5. Write summary to `school-summary-YYYY-MM-DD.md`
6. Optionally mark emails as read

#### School sender matching

`context.toml` uses a `school_from` field per child — a list of sender addresses/domains to match. The skill builds a single query per school using Gmail's `{...}` OR syntax:

```
{from:synapseschool.org from:synapseschoolorg.myenotice.com} is:unread newer_than:7d
```

Gmail's `from:` matches at the word level (not substring), so multiple entries are needed when a school sends from multiple domains (e.g. main domain + newsletter provider).

#### Summary formatting (`prompt.md`)

Enforces a strict structure: Action Items → Calendar (single chronological source of truth for all dates) → Per-Child Notes (no dates, only context) → Emails Included (markdown table). This prevents date duplication and gaps that occurred when sections were free-form.

#### Installing skills

`skills/` is a source directory (modeled after [anthropics/skills](https://github.com/anthropics/skills)). Users install by symlinking:

```bash
ln -s /Users/anna/src/gmail-assistant/skills/school-emails ~/.claude/skills/school-emails
```

Run `/reload-plugins` if Claude is already running. Invoke with `/school-emails`.

#### `/order-emails` skill (future iteration)
Tracking purchases (shopping, food delivery, subscriptions, bills). Will follow the same pattern. Deferred until school-emails is mature.

# Plan: v0.4 — Skills Restructure

## Context

The project currently bundles everything into one MCP server: Gmail actions (mark as read, fetch body), search/listing, caching, and the school email summarization workflow. The summarization logic (personal context loading, prompt instructions, orchestrating cache + file paths) lives in MCP tool code, but it's really a Claude workflow — not a programmatic action. This restructure separates concerns: the MCP server handles Gmail actions, and Claude Code skills handle email management workflows.

## Background

### Email caching (v0.3)

The v0.3 release (see `docs/v0.3-email-cache.md`) introduced `gmail_cache.py` to solve MCP size limits. Emails are saved as local markdown files in `emails/YYYY/MM/{message_id}.md` with YAML frontmatter (id, subject, from, date) + body. Key functions:
- `save_email(service, message_id) -> Path` — fetch from Gmail API if not cached, write to disk
- `ensure_cached(service, message_ids) -> list[Path]` — batch cache, returns file paths
- `find_cached(message_id) -> Path | None` — glob lookup
- `read_cached(message_id) -> str | None` — read body (strips frontmatter)

Both `gmail_fetch_body` (MCP tool) and the skills rely on this cache. The `gmail_fetch_body` tool calls `save_email` then `read_cached` transparently. Skills use the cached file paths to have Claude read emails directly via the Read tool — no MCP size limit.

### Email markdown cleanup (v0.2b — not yet implemented)

`docs/v0.2b-email-cleanup.md` plans an HTML preprocessing step in `gmail_helpers.py` to shrink cached emails from ~24KB to ~11KB by unwrapping layout tables and shortening tracking URLs. This runs in `body_to_markdown()` for all emails. Not blocking this restructure — can be applied before or after. If done before, cached school emails will be cleaner when skills read them. If done after, existing caches can be deleted and re-fetched.

## Architecture

### MCP server (gmail actions only)
Tools that do things to Gmail that Claude can't do natively:
- `gmail_mark_as_read` — mark message read (existing)
- `gmail_fetch_body` — fetch + cache email body as markdown (existing)
- `gmail_list_messages` — search Gmail by query (existing)
- (Future) `gmail_archive` — archive a message
- (Future) `gmail_delete` — trash a message

Remove `gmail_summarize_school_emails` from the MCP server — that workflow moves to a skill.

### Skills (email management workflows)

#### `/school-emails` skill
Orchestrates: search school emails → cache them → read cached files → summarize with personal context.

```
skills/school-emails/
├── SKILL.md              # Skill definition + instructions
├── prompt.md             # Summary instructions (moved from prompts/school_summary.md)
└── context.toml          # Children, grades, teachers, etc. (moved from project root)
```

SKILL.md frontmatter:
```yaml
---
name: school-emails
description: Summarize recent school emails with personal context about your children
allowed-tools: "mcp__gmail-assistant__gmail_list_messages mcp__gmail-assistant__gmail_fetch_body mcp__gmail-assistant__gmail_mark_as_read Read Write Glob"
---
```

SKILL.md body instructs Claude to:
1. Read skill-local `context.toml` for personal context (child names, grades, teachers, etc.)
2. Call `gmail_list_messages` with school domain queries
3. Call `gmail_fetch_body` for each message (caches automatically)
4. Read cached email files
5. Read `prompt.md` for summary instructions
6. Write summary to `school-summary-YYYY-MM-DD.md`
7. Optionally mark emails as read

#### Installing skills

`skills/` is a source directory (modeled after [anthropics/skills](https://github.com/anthropics/skills)). Users install by symlinking into Claude Code's discovery path:

```bash
# Clone the repo, then symlink the skill
ln -s /path/to/gmail-assistant/skills/school-emails ~/.claude/skills/school-emails
```

Run `/reload-plugins` if Claude is already running. Invoke with `/school-emails`.

Future option: package as a plugin marketplace for `/plugin marketplace add` + `/plugin install` workflow.

#### `/order-emails` skill (future iteration)
Tracking all purchases (shopping, food delivery, subscriptions, bills). Will follow the same pattern as school-emails with its own `context.toml` for label preferences and categories. Deferred until school-emails is working.
